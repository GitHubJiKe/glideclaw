既然你追求的是一个**轻量、高性能且具备极客感**的本地 AI 助手，我们跳过那些沉重的企业级架构，直接进入以 **Bun** 为核心、**Local-first** 为灵魂的“窄窗口”架构设计。

这套方案我称之为 **"Rolling Context Engine" (滚动上下文引擎)**。

---

## 1. 核心架构全景图

整个系统分为三层：**感知层**（输入）、**策略层**（逻辑处理与记忆裁减）、**执行层**（LLM 与本地动作）。

### 模块化分层设计

1. **CLI / Daemon 入口 (Bun):** 负责监听用户指令或系统钩子。
2. **Context Manager (策略核心):** 核心逻辑所在。负责计算当前时间戳，查询 SQLite 提取 $N$ 天内的记录。
3. **Token Budgeter (成本卫士):** 实时计算当前窗口的 Token 消耗，确保不超过模型限制或用户设定的阈值。
4. **Vector Lite (可选扩展):** 使用简单的向量索引（如 `hnswlib` 的 JS 封装）存储过期但重要的“元数据”。

---

## 2. 存储方案：基于 `bun:sqlite` 的滑动窗口

这是 `GlideClaw` 实现“轻量”的关键。不同于传统的全量存储，我们采用**物理删除+逻辑摘要**的模式。

### 数据库表结构 (Schema)

* **`conversations`**: 存储原始对话。
* `id`, `role`, `content`, `tokens`, `timestamp`


* **`summaries`**: 存储“日结”摘要。
* `date`, `summary_text` (每天结束时，AI 自动生成该日的极简总结，约 100-200 token)



### 滚动逻辑

每次启动或对话前，执行一个简单的 SQL 维护：

```sql
-- 这里的 7 可以是用户配置的 1-30
DELETE FROM conversations WHERE timestamp < datetime('now', '-7 days');

```

---

## 3. 核心流转过程 (Pipeline)

当用户输入一个问题时，`GlideClaw` 的内部路径如下：

1. **时区对齐：** 获取当前本地时间。
2. **上下文装载：**
* 从 `conversations` 读取最近 $N$ 天的原文。
* （可选）从 `summaries` 读取 $N$ 天之前、30 天之内的日结摘要（作为背景知识，极省 Token）。


3. **Prompt 组装：**
```markdown
[System Role: 你是一个极简主义助手...]
[Context History (Last 7 Days): ...]
[User Question: ...]

```


4. **流式输出：** 利用 Bun 的 `fetch` 配合 `ReadableStream` 实时渲染结果到终端。
5. **异步持久化：** 对话结束，自动计算 Token 并存入 SQLite。

---

## 4. 关键技术选型清单

| 维度 | 推荐工具 | 理由 |
| --- | --- | --- |
| **运行时** | **Bun** | 启动快，内置 SQLite 和原生 Fetch，极致的 IO 性能。 |
| **数据库** | **`bun:sqlite`** | 零配置，单文件存储，读写延迟极低。 |
| **配置管理** | **`Zod`** | 用于校验用户定义的 `config.json`（如设置 1-30 天的限制）。 |
| **LLM 通讯** | **OpenAI SDK (Node 兼容)** | 虽然是 Node 版，但在 Bun 下跑得飞快。 |
| **终端 UI** | **`Clack` 或 `Inquirer**` | 打造漂亮、现代化的 CLI 交互界面。 |

---

## 5. 为什么这套方案能“丰俭由人”？

* **俭：** 用户设置 1 天记忆。每次对话只带入当天的上下文，Token 消耗极小，适合处理临时任务。
* **丰：** 用户设置 30 天记忆。虽然 Token 变多，但通过“日结摘要”机制，依然比传统的 OpenClaw 节省约 **60% - 80%** 的成本。
* **定制化：** 你可以为不同的“助手”设置不同的窗口。比如“代码助手”设置 3 天，“生活日记助手”设置 30 天。

---

## 6. 实现建议：第一步做什么？

我建议你先不要写复杂的 UI，先用 Bun 写一个 **CLI 原型**，实现以下逻辑：

1. 用 `bun:sqlite` 建表。
2. 写一个 `add_message` 和 `get_window_messages` 的函数。
3. 测试一下自动删除超过 7 天数据的功能。

如果你觉得这个架构方向没问题，**我们要不要深入探讨一下那个“日结摘要（Daily Summary）”的自动生成算法？** 它是平衡“记忆长度”和“Token 成本”的最强补丁。
